## 1. 현황 진단 요약

### 1.1 핵심 문제

| 구분 | 문제 | 현상 |
|------|------|------|
| 구조 | 기능별 조직 | P10 전체를 조망하는 책임자 없음 |
| 구조 | 겸임 리더십 | 이일섭(영업+IoT), 진원경(AI+전략기획) |
| 제품 | P10 미안정화 | 3년+ 개발, 시장 안착 못함 |
| 제품 | 기획-개발 분리 | 전략기획실 ↔ 연구소 물리적 분리 |
| 역량 | AI 과소 | 5명, 소장 겸임 |
| 역량 | ITSM 방치 | 연구소가 ITSM 개발을 담고 있지 못함 |

### 1.2 문제의 근본 원인

- 제품별 책임/권한 불명확
- 영업이익 기반 인센티브 구조 부재
- 의사결정 레이어 과다
- 겸임으로 인한 집중력 분산

## 2. 독립 P&L 사업 단위

### 2.1 설계 원칙

- 각 사업 단위는 **독립적인 P&L (손익계산)**을 가짐
- 영업이익 기반 **인센티브 구조** 명확화
- 제품 책임자가 해당 사업의 의사결정권 보유

### 2.2 사업 단위 정의

| 사업 단위 | P&L | 제품 범위 | 비고 |
|----------|-----|----------|------|
| **P10 사업부** | ✅ 독립 | EMS + ITSM (번들/개별 무관) | 핵심 사업 |
| **P8 유지보수** | ✅ 별도 수익센터 | P8 기존 고객 | 점진 축소 |
| **IoT 사업부** | ✅ 독립 | 와이어로프 IoT | 신성장 사업 |
| **보안 사업부** | ✅ 독립 | 보안 신제품 | 신설, 물리적 분리 |

### 2.3 P10 제품 판매 구조

```
P10 판매 조합 (모두 P10 사업부 P&L)
├── EMS만 판매
├── ITSM만 판매
├── EMS + ITSM 전체 번들
└── EMS + ITSM 일부 기능
```

→ EMS/ITSM을 별도 P&L로 분리하지 않고, **P10 전체를 하나의 사업 단위**로 운영

## 3. 조직 구조

### 3.1 조직도

```
회장
│
├── P10 사업부 ─────────── 사업부장 (독립 P&L)
│   │
│   ├── [개발조직]
│   │   ├── EMS 개발팀 (기획+FE+BE+디자인)
│   │   ├── ITSM 개발팀 (기획+FE+BE+디자인)
│   │   └── Platform 개발팀 (공통 기능)
│   │
│   └── [사업조직]
│       ├── 영업팀
│       ├── 기술지원팀
│       └── 컨설팅팀
│
├── P8 유지보수 ─────────── 조직장 (별도 수익센터)
│   └── P8 개발 + 기술지원
│
├── IoT 사업부 ──────────── 사업부장 (독립 P&L)
│   └── IoT팀 (개발 + 사업)
│
├── 보안 사업부 ─────────── 사업부장 (독립 P&L, 물리적 분리)
│   └── 보안팀 (개발 + 사업)
│
├── AI연구소 ────────────── 연구소장 (전사 AI 지원)
│   ├── AI Platform팀
│   └── AI 제품지원 (P10 담당, 보안 담당 등)
│
├── 신사업탐색 조직 ──────── 조직장 (신시장 개척 전담)
│
└── 경영지원본부
    ├── 품질혁신팀
    └── 경영지원팀
```

### 3.2 핵심 설계 포인트

| 항목 | 설계 | 이유 |
|------|------|------|
| P10 사업부 | 개발 + 사업을 **같은 사업부장 아래** | 책임/권한 일치, 인센티브 명확 |
| P8 유지보수 | **별도 조직** 분리 | 신제품(P10)과 레거시 분리 |
| ITSM 개발팀 | P10 사업부 개발조직으로 이동 | 연구소 통합, ITSM 방치 해소 |
| 개발조직 내 기획/디자인 | **배치 가능** | 기획-개발 통합 |
| 보안 사업부 | **물리적 분리** | 별도 조직으로 집중 |
| 경량 구조 | 회장 → 사업부장 → 팀장 (3단계) | 의사결정 속도 |

### 3.3 현재 → 신규 인력 이동

| 현재 소속 | 신규 소속 | 비고 |
|-----------|-----------|------|
| 연구1팀 (P8 전담) | P8 유지보수 + P10 개발조직 일부 | P8 축소에 따라 전환 |
| 연구2팀 (K8s/APM) | P10 EMS 개발팀 | - |
| 연구3팀 (대시보드/디자인) | P10 Platform 개발팀 + 디자이너 배치 | - |
| 연구4팀 (P10 공통) | P10 EMS/Platform 개발팀 | - |
| 사업지원팀 ITSM개발 (5~6명) | **P10 ITSM 개발팀** | 핵심 이동 |
| AI연구소 (5명) | AI연구소 (확대) | - |
| 전략기획실 기획자 (2명) | P10 개발조직 내 기획 | - |

## 4. AI 조직 운영

### 4.1 운영 원칙

| 항목 | 결정 | 이유 |
|------|------|------|
| AI 인력 소속 | **전원 AI연구소** | 역량 집중, 표준화 |
| 조직 형태 | **중앙 집중** (분산 ❌) | 경량 문명, AI에 힘 실어주기 |
| 업무 지시 | **연구소장** | 명확한 지휘체계 |

### 4.2 AI연구소 구조

```
AI연구소 ─── 연구소장 (경영진급, 회장 직보고)
│
├── AI Platform팀 (2~3명)
│   • MLOps, 인프라
│   • AI 기술 표준
│   • 공통 모델/라이브러리
│
└── AI 제품지원 (5~10명)
    • P10 담당: 2~3명 (AIOps 등)
    • 보안 담당: 2~3명 (보안 AI)
    • 기타 지원: 필요시
```

### 4.3 사업부 협업 방식

| 항목 | 방식 |
|------|------|
| 사업부 회의 참석 | **담당자가 참석** (연구소장 ❌) |
| 우선순위 조율 | 연구소장 ↔ 사업부장 협의 |
| 기술 의사결정 | 연구소장 권한 |

**연구소장 역할**:
- 전사 AI 전략 수립
- AI 기술 표준/거버넌스
- AI 인력 채용/평가
- 경영진 회의 참석
- 중대 기술 의사결정

**담당자 역할**:
- 해당 사업부 회의 참석
- 제품 AI 기능 개발
- 연구소장에게 주간 보고

### 4.4 AI 유지보수 정책

| 유지보수 유형 | 담당 | 비고 |
|--------------|------|------|
| API 버그 수정 | 사업부 BE 개발자 | 일반 코드 수준 |
| 설정값 변경 | 사업부 BE 개발자 | 임계치 등 |
| 모델 재학습 | **AI연구소** | AI 전문 영역 |
| 파라미터 튜닝 | **AI연구소** | AI 전문 영역 |
| 성능 저하 분석 | **AI연구소** | AI 전문 영역 |
| 모델 구조 변경 | **AI연구소** | AI 전문 영역 |

> 시계열 알고리즘, 딥러닝 모델 모두 모델 관련 유지보수는 **AI 전문가만 가능**
