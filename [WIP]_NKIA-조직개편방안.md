# NKIA 조직개편 방안

---

## 0. 메타사고: 왜 조직을 개편해야 하는가?

### 사고의 출발점

조직 개편을 논하기 전에, **"왜 지금 개편이 필요한가?"**를 구조적으로 점검한다.

```
[메타사고 프레임]

Q1. 현재 조직은 어떤 목표를 위해 설계되었는가?
  → Polestar8 기반 EMS 제품 개발·판매·유지보수

Q2. 그 목표는 아직 유효한가?
  → 부분적으로만 유효. EMS는 레거시 시장이며 성장 한계에 도달

Q3. 새로운 목표는 무엇인가? ⚠️ [핵심 논의 필요]

  [현실 인식]
  • 기존 EMS/ITSM **온프레미스 시장**은 확장성이 없고, 점차 축소될 것으로 예상
  • AI 코딩 능력 확산으로 오라클, 세일즈포스 같은 SaaS SW 대기업조차 위협받는 상황
  • 우리는 솔루션 회사인데, 이 환경에서 어떻게 생존하고 성장할 것인가?

  [클라우드 시장 현실 분석]
  • 클라우드 EMS(SMS/NMS/DPM): AWS CloudWatch, Azure Monitor 등 벤더가 기본 제공 → 별도 시장 없음
  • 클라우드 APM/K8s: 시장 존재하나 Datadog, New Relic, Dynatrace 등 글로벌 플레이어가 장악
  • 클라우드 ITSM: ServiceNow가 지배하는 레드오션
  • 결론: 클라우드 시장에서 EMS 제품으로 경쟁하기 어려움

  [새로운 시장 기회 후보] ❓ 논의 필요
  • AIOps: AI 기반 이상탐지, 근본원인분석(RCA), 자동 조치 → 기존 모니터링 역량 + AI 결합
  • 하이브리드 모니터링: 온프레미스 + 클라우드 통합 관제 → 국내 기업 니즈 존재
  • FinOps: 클라우드 비용 최적화 → 신규 시장, 진입 가능성?
  • 보안 모니터링/SIEM: 현재 보안 TF로 검토 중
  • 옵저버빌리티 플랫폼: 메트릭 + 로그 + 트레이스 통합

  [전략적 질문]
  • IoT 사업부를 제외하고, 엔키아는 어디서 새로운 매출을 창출할 것인가?
  • 어떤 시장을 타겟으로 해야 하는가?
  • 어떤 전략으로 그 시장을 공략할 것인가?
  • 이를 실행할 수 있는 조직 구조는 무엇인가?

  [AI 인력 조직화 문제] ⭐ 핵심 관심사
  • AI 인력은 별도 조직(AI연구소)으로 유지해야 하는가?
  • 각 제품팀에 분산 배치(Embedded)해야 하는가?
  • 하이브리드(Core + Embedded) 모델이 적합한가?
  • AI가 제품의 차별화 요소가 될 수 있는가, 아니면 내부 생산성 도구에 그칠 것인가?

  → 이 질문들에 대한 답이 조직 개편의 방향을 결정함

Q4. 현재 조직 구조로 새 목표를 달성할 수 있는가?
  → 불가. 3년째 P10 미안정화가 그 증거

Q5. 그렇다면 무엇을 바꿔야 하는가?
  → 이 문서의 본론
```

### 개편의 본질적 질문

> **"NKIA는 EMS 유지보수 회사인가, IT 운영 보증 플랫폼 기업인가?"**

미션 문서 LEAP FORWARD는 후자를 선언하고 있다. 그러나 현재 조직 구조는 전자에 최적화되어 있다. **미션과 조직 사이의 괴리**가 개편의 근본 이유다.

## 1. 현황 진단 (MECE 분석)

NKIA의 현재 문제를 **상호배제·전체포괄(MECE)** 원칙으로 분류한다.

### 1.1 MECE 문제 분류

```
NKIA 조직 문제
├── A. 구조적 문제 (Structure)
│   ├── A1. 기능별 조직 vs 제품별 필요
│   ├── A2. 겸임에 의한 리더십 분산
│   └── A3. 의사결정 레이어 과다
│
├── B. 제품·기술 문제 (Product)
│   ├── B1. P10 개발 3년째 미안정화
│   ├── B2. 기획-개발 분리 (전략기획실 ↔ 연구소)
│   └── B3. FE 역량 공백 (NTQ 외주 의존)
│
├── C. 역량·인재 문제 (Capability)
│   ├── C1. AI 조직 과소 (5명)
│   ├── C2. ITSM 개발 역량의 조직적 방치
│   └── C3. 기술 오버엔지니어링 경향
│
└── D. 문화·프로세스 문제 (Culture)
    ├── D1. 사일로 간 비협조 (연구소가 ITSM 거부)
    ├── D2. 제품 오너십 부재
    └── D3. 학습·피드백 문화 미약
```

```
전사 체질 전환
│
├── 1. P10 조기 안정화 및 시장 안착

│   ├── 4.1 위계 단계 축소 (5단계 → 3단계)

└── 5. 조직 문화 전환
    ├── 5.1 제품 오너십 문화 확립
    ├── 5.2 Learn-Unlearn-Relearn 학습 체계
    └── 5.3 크로스펑셔널 협업 문화

## 3. 개편 방안: 제품 중심 Squad 체제

### 3.1 설계 원칙

| 원칙 | 설명 | 이론적 근거 |
|------|------|-------------|
| **제품 중심** | 기능별 팀 → 제품/미션별 팀 | 스포티파이 Squad, Amazon Two-Pizza |
| **자기 완결** | 기획+디자인+FE+BE+QA를 Squad 내 통합 | AX조직 "부서 간 장벽 파괴" |
| **오너십** | 각 Squad에 Product Owner 1명 | Netflix Informed Captain |
| **경량 위계** | 본부장 레이어 축소, Squad Lead가 의사결정 | Bayer DSO, 송길영 경량문명 |
| **AI 내재화** | 모든 Squad에 AI 활용 의무화 | McKinsey Agentic Organization |

### 3.2 신규 조직도 (안)

```
대표이사 / 김영부 회장
│
├── Product Division (제품본부) ─── CPO (신설, 또는 기존 임원 전환)
│   │
│   ├── P10 Tribe (트라이브)
│   │   ├── [Squad] P10-EMS ── 기획1+FE2+BE4+QA1 (8명)

## 11. 핵심 메시지

### 한 문장 요약

> **"기능별로 나뉜 연구소 체제를 제품 중심 Squad로 전환하고, AI를 전사에 내재화하며, 의사결정 레이어를 3단계로 축소한다."**

### 개편의 3대 축

```
┌─────────────────────────────────────────────────────────┐
│                                                         │
│   1. 구조: 기능별 팀 → 제품별 Squad                      │
│      "만드는 사람이 기획하고, 기획하는 사람이 같이 앉는다" │
│                                                         │
│   2. AI: 분리된 5명 → 확대 + 전사 Embedded               │
│      "AI는 별도 조직이 아니라 모든 제품의 DNA"             │
│                                                         │
│   3. 속도: 5단계 위계 → 3단계 경량 구조                   │
│      "결정이 빠른 조직이 살아남는다"                      │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

### 코닥의 교훈을 잊지 말 것

> 코닥은 1975년 세계 최초로 디지털 카메라를 만들었다. 기술이 없어서 망한 것이 아니다. **조직이 변하지 않아서 망했다.** NKIA도 AI 기술과 모니터링 역량을 보유하고 있다. 그러나 이 역량을 살리려면 **조직이 먼저 바뀌어야 한다.**
